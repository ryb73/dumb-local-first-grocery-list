# Server Migration Plan: From Simulated to Real Server

## Overview

This document outlines the plan to refactor the current simulated server architecture (where both client and server use local SQLite databases in OPFS) to a real Express server with SQLite, while maintaining the existing sync algorithm and minimal architectural changes.

## Current Architecture Analysis

### Client-Side Components (staying in `front-end/`)
- **UI Components**: All React/Solid components in `src/components/`
- **Client Database**: Local SQLite database via OPFS
- **Client Sync Logic**: `src/sync/client/`, `GroceryList.tsx`
- **Operation Logging**: Client-side operation capture and logging

### Server-Side Components (to be moved to real server)
- **Server Sync Logic**: `src/sync/server/`
  - `sync.ts` - Main sync endpoint logic
  - `apply-client-changes.ts` - Server-side operation application
  - `operations.ts` - Server operation queries
  - `migration-state.ts` - Server migration state management
  - `database.ts` - Server database connection (currently simulated)

### Shared Components (to be moved to shared library)
- **Database Schema & Types**:
  - `db.d.ts` - Main database type definitions (note that this is auto-generated by `kysely-codegen`)
  - `operation-log-db.d.ts` - Operation log type definitions (note that this is auto-generated by `kysely-codegen`)
  - `src/db/merged-db.d.ts` - Combined database types
- **Migration System**:
  - `src/db/migrations/` - Main database migrations
  - `src/db/operation-log/migrations/` - Operation log migrations
  - `src/db/init.ts` - Database initialization logic
- **Operation Types & Logic**:
  - `src/operation-logging/operation-types.ts` - Operation type definitions
  - `src/operation-logging/apply-operation.ts` - Operation application logic
  - `src/operation-logging/reverse-operation.ts` - Operation reversal logic
  - `src/operation-logging/resolve-conflict.ts` - Conflict resolution logic
  - `src/operation-logging/rebase.ts` - Rebase algorithm
- **Shared Types**:
  - `src/types/schemas.ts` - Shared type definitions
- **Challenge**: sqlocal (OPFS) vs better-sqlite3 (filesystem) differences
- **Solution**: Use Kysely's SQLite dialect for both, abstract the driver creation

## Proposed Architecture

### 1. Three-Project Structure

```
grocery-list/
├── front-end/           # Client application
├── back-end/            # Server application
└── shared/              # Shared library
```

### 2. Server Application (`back-end/`)

#### Technology Stack:
- **Node.js + Express** (simple HTTP server)
- **SQLite + better-sqlite3** (same as current, but on server filesystem)

#### Package Structure:
```
back-end/
├── package.json
└── src/
    ├── database/
    │   └── connection.ts     # Server database connection
    ├── sync/
    │   ├── ...               # Server-side sync logic
    └── server.ts             # Main server entry point
```

#### Key Features:
- **HTTP API**: Single `/sync` endpoint (matches existing server function interface)
- **SQLite Database**: Uses better-sqlite3 on server filesystem instead of OPFS
- **Migration Management**: Same migration system, runs on server startup
- **Error Handling**: HTTP error responses for network layer
- **Minimal Changes**: Existing server logic moves with minimal modification

### 3. Updated Client (`front-end/`)

#### Changes Required:
- **Remove Server Simulation**: Delete `src/sync/server/` directory. In ParallelGroceryLists.tsx, both GroceryList instances should sync with the same server but use a separate database.
- **HTTP Client**: Replace direct server function calls with HTTP requests

## Migration Steps

### Phase 1: Create Shared Library

1. **Create shared package structure**
   ```bash
   mkdir -p grocery-list/shared/src/{database,operations,sync,types}
   cd grocery-list/shared
   npm init -y
   ```

2. **Move shared code to shared library**:
   - Database types and schemas
   - Migration definitions and runners
   - Operation types and logic
   - Conflict resolution and rebase algorithms

3. **Set up build system**:
   - TypeScript compilation
   - Type definitions export

4. **Update package dependencies**:
   - Add necessary dependencies (kysely, zod, etc.)

### Phase 2: Create Server Application

1. **Initialize server project**:
   ```bash
   mkdir -p grocery-list/back-end
   cd grocery-list/back-end
   npm init -y
   ```

2. **Set up server framework**

3. **Move server logic**:
   - Port sync logic from `src/sync/server/`
   - Adapt for HTTP endpoints
   - Update database connection logic

4. **Implement HTTP API**:
   - POST `/sync` endpoint
   - Request/response validation
   - Error handling and status codes

### Phase 3: Update Client Application

1. **Remove server simulation**:
   - Delete `src/sync/server/` directory
   - Remove server database references

2. **Implement HTTP client**:
   - Create HTTP client for sync operations
   - Handle network errors
   - Update sync logic to use HTTP calls
